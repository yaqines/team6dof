Tight Integration
Detection output feeds directly into flight commands. There is no manual relay step between seeing a target and acting on it.
Redundant by Design
Every layer has a defined fallback. If a subsystem degrades mid-mission, the system transitions to a safe mode without operator input.
Live Situational Awareness
Telemetry, video, and detection results all reach the GCS in real time so the operator always knows what the system is seeing and doing.
AI and Autonomy

From pixels to decisions.

Our detection pipeline takes raw imagery and turns it into actionable outputs. Each stage is designed with latency and accuracy in mind, and the whole thing runs onboard during flight.

01
Image Capture
A downward-facing camera captures frames at set intervals across the survey area.
02
Preprocessing
Frames are resized, normalized, and tagged with GPS and IMU data for accurate spatial mapping.
03
Detection Model
A fine-tuned YOLO model classifies SUAS targets by shape, color, and alphanumeric character.
04
Localization
Pixel positions are converted to GPS coordinates using camera intrinsics and altitude data.
05
Decision Output
Confirmed targets are queued for payload drop, waypoint update, or operator review.
Decision Logic

Confidence-gated actions

Every detection goes through a confidence threshold before anything happens. Anything below that threshold gets flagged for the operator rather than acted on automatically. This keeps false positives from triggering a drop.

The operator stays in the loop throughout. Autonomy here means handling the routine decisions, not removing human oversight.

Mapping and Localization

Knowing where targets are

Target GPS coordinates are computed from pixel position, camera field of view, altitude, and heading. Results appear on the live map in the GCS as detections come in.

We use multi-frame averaging to improve accuracy across overlapping passes of the same area.

Hardware and Airframe

Built to carry the mission,
not to impress.

Hardware decisions here come from requirements, not preferences. We picked a configuration that gives us the lift, redundancy, and flexibility the mission actually needs, and we did not add anything beyond that.

Airframe
We chose a hexacopter configuration because it gives us motor redundancy and enough lift margin for the payload. The frame is sized around the system requirements, not the other way around.
Payload Mechanism
The drop system is actuated on command from the flight controller. Release timing accounts for velocity and altitude so the payload lands where it needs to.
Structural Priorities
Weight, rigidity, and repairability were the main design constraints. Parts that break in the field need to be replaceable, not just designed to never fail.
Flight Control and GCS

Controlled flight.
Clear situational awareness.

Our flight stack is built for reliability and transparency. The autopilot handles the routine workload and the GCS keeps the operator informed at every step.

Autopilot and Navigation
The flight controller fuses data from multiple sensors to maintain a stable, accurate picture of the aircraft's position and attitude throughout the mission. Waypoint following and autonomous behaviors run on top of this foundation.

Mission Strategy

Reliable first.
Everything else second.

We plan around what we can actually execute, not what looks ambitious on paper. Complexity gets added when there is a clear reason for it, and not before.

Search and Detection
The drone runs a survey pattern over the search area with enough overlap to ensure full coverage. Detection results from each pass are aggregated and cross-checked before we act on them.
Target Engagement
Once a target is confirmed, the drone returns to it at the right altitude and releases the payload at the calculated moment. Speed and altitude both factor into the timing.
Contingency Handling
Low confidence means re-survey, not a guess. Every edge case in the mission logic has a defined fallback so the system does not end up in an undefined state mid-flight.
How we work
Use proven approaches before trying novel ones

Trade-offs we accepted
Hexacopter is heavier than a quad, but more resilient
Conservative detection thresholds mean fewer false drops
Re-survey costs flight time but improves accuracy
Manual confirmation adds a step but keeps the operator informed
Single camera keeps things simple at a cost to redundancy
Testing and Validation

We test until it is boring.

Each subsystem gets validated on its own before it touches anything else. Then the whole thing gets tested together, repeatedly, until the behavior is predictable.

01
Software-in-the-Loop
Full mission simulation before any hardware is involved. Flight paths, detection events, and drop sequences all run in a virtual environment first.
02
Hardware-in-the-Loop
Real flight controller, simulated environment. This catches firmware and interface issues before they show up in the air.
03
Bench Integration
All subsystems assembled and talking to each other, but not flying. Power, data, and comms links are verified end to end.
04
Hover and Stability
Actual flight tests starting with tethered hover. We tune the controller and review flight logs before expanding the envelope.
05
Full Mission Rehearsal
An end-to-end run with real targets in a controlled field. We measure drop accuracy and refine until the results are consistent.
06
Repeatability Gate
We require five clean consecutive rehearsals before clearing the system for competition. Any failure means we investigate before continuing.
